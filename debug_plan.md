# VEX 机器人自动程序数据回传与调试优化计划

## 一、 代码部署策略（如何优雅地回传数据）

1. **多线程/异步任务分离**
   - **计划**：不要在核心的 PID 计算循环中直接调用 `printf` 或 `std::cout`，这会带来严重的 I/O 延迟，破坏 PID 循环的固定周期（通常是 10ms）。
   - **做法**：单独创建一个低优先级的后台任务（Task/Thread），专门负责数据回传。通过全局变量或结构体将 PID 循环中的数据传递给这个打印任务。

2. **使用统一的数据格式（如 CSV 格式）**
   - **计划**：将回传数据格式化为纯文本的逗号分隔值（CSV），例如 `Time,Target,Actual,Error,Output`。
   - **优势**：极大地降低解析难度，方便后续直接复制到 Excel 中，或者通过 Python 脚本、串口绘图工具进行实时波形可视化。

3. **宏定义开关（Debug Mode）**
   - **计划**：使用 `#define DEBUG_PID_TURN` 等宏定义来包裹打印代码。
   - **优势**：在日常调车时开启宏进行高频打印；在正式比赛前，只需注释掉宏定义即可一键关闭所有打印，避免占用 V5 大脑的 CPU 资源。

## 二、 核心回传数据选择（该看哪些数据）

为了能够精准定位是“参数没调好”还是“算法有逻辑缺陷”，建议每一帧回传以下数据组合：

1. **基础状态数据**
   - `Timestamp`：系统运行时间（毫秒），作为数据图表的 X 轴，检查循环周期是否稳定。
   - `State/Step ID`：当前处于自动程序的哪一个步骤（例如：1代表第一段直行，2代表转向）。用于排查**算法缺陷**（比如程序卡在某一步无法进入下一步，或者提前退出了循环）。

2. **PID 调参核心数据**
   - `Target (SetPoint)`：目标值（目标编码器 tick 数或目标 IMU 角度）。
   - `Actual (Sensor)`：实际传感器读数。
   - `Error`：当前误差（Target - Actual），用于观察是否收敛。
   - `Derivative`：误差变化率（当前误差 - 上次误差），用于观察 D 项是否有效抑制震荡。

3. **输出与限幅数据**
   - `Motor Power/Voltage`：PID 计算后最终给到电机的电压或速度指令。用于观察是否频繁触发满功率（导致饱和失控）或输出过小（导致电机啸叫但不动）。

## 三、 调试流程优化（如何成倍提升调参效率）

单纯看终端里疯狂滚动的数字是很难调好 PID 的，需要对流程进行以下升级：

1. **将回传数据交由 AI 分析与反馈**
   - **方案**：无需配置额外的绘图工具，将终端中打印出来的 CSV 格式数据块直接复制并发送给我（AI）。
   - **目的**：我会直接帮你分析数据中的**超调量（Overshoot）**、**震荡频率（Oscillation）**和**稳态误差（Steady-state Error）**，并直接给出下一步的调参建议（如增加/减小某个参数）及算法逻辑诊断。

2. **复用已有的标准化测试死函数**
   - **方案**：直接使用 `void.h` 中已定义好的 `test` 系列函数（如 `test_straight()`, `test_turn()`, `test_gyro_pd()` 等）来进行基准测试。
   - **流程**：先利用 `test_straight()` 调底盘直行 PID，再用 `test_turn()` 调原地转向 PID。每次修改参数后，执行这些死函数，保证机器人的起始位置和测试动作必须完全一致，排除场地摩擦力和电池电量下降带来的干扰。

3. **算法缺陷排查专项**
   - **方案**：如果你怀疑是退出条件（Exit Conditions）写得不对（例如 `while(error > tolerance)` 死循环），可以在回传数据中加入 `SettledTime`（保持在误差范围内的时间）和 `Timeout` 标志位。这能帮你明确知道机器人是因为“达到目标”停下来的，还是因为“超时强制退出”停下来的。
